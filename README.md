# 中文裁员协商指南

一个面向中文职场裁员协商场景的通用 AI Agent Skill / Playbook。仓库名保留为 `layoff-negotiation-cn`，方便在不同 Agent 或工具中引用。

它不是法律意见，也不是鼓励所有人都去硬刚公司。它更像一份“被突然通知裁员时，先别把自己谈崩”的行动手册：帮助普通人在压力很大的时候，先稳住、算清楚、留好证据、看懂 HR/ER 话术，再决定是签、谈、等单方解除，还是准备仲裁。

## 这个 Skill 想解决什么问题

很多人第一次被通知裁员时，通常会同时遇到这些问题：

- 刚被通知裁员，很快就收到协商离职电子签，不知道能不能签。
- HR 说“这是最优方案”“这是特殊申请”“现在不签后面就不好说了”。
- 不知道 N、N+1、2N 怎么算，也不知道值不值得继续谈。
- 担心背调、离职证明、社保、公积金、找工作衔接。
- 协商谈不拢后，被贴上“不符合预期”“绩效不合格”之类标签。
- 不知道证据应该拍什么、怎么拍、哪些不能碰。
- 一边要找工作，一边还要继续上班，情绪和压力都很重。

这个 Skill 希望把这些散落的经验整理成一套可以被 AI 调用的流程，让遇到类似问题的人少踩坑。

## 怎么让 Agent 使用

把你的当前处境直接告诉 Agent，例如：公司怎么通知你的、有没有让你签字、给了什么补偿、工龄多久、是否出现绩效争议、是否还在正常上班、你最担心什么。

Agent 应按本 Skill 的流程帮你整理：

- 当前路径和风险点。
- 需要保留的证据。
- 补偿测算思路。
- HR/ER 回复话术。
- 是否需要律师确认的问题。
- 接下来一个安全动作。

建议一次只处理一个具体问题，例如“这个协议能不能签”“绩效申诉会怎么说”“HR 说影响背调怎么回”“我应该拍哪些证据”。这样 Agent 更容易给出可执行答案。

## 内容结构

- `SKILL.md`：Skill 的主入口和工作流。
- `references/first-24-hours.md`：被通知裁员后的前 24 小时。
- `references/hr-er-scripts.md`：HR/ER 沟通话术和风险识别。
- `references/performance-dispute.md`：绩效不合格、PIP、绩效申诉会应对。
- `references/evidence-checklist.md`：证据清单和取证边界。
- `references/compensation-and-arbitration.md`：N、N+1、2N、补偿测算和仲裁现实成本。
- `references/economic-layoff.md`：经济性裁员程序、优先留用和重新招用规则。
- `references/unemployment-benefits.md`：失业金申领、社保医保和公积金衔接。
- `references/non-compete.md`：竞业限制、竞业补偿和解除边界。
- `references/separation-certificate.md`：离职证明内容、法律依据和重开话术。
- `references/work-and-job-search.md`：在岗期间考勤、产出、找工作边界。
- `references/emotional-reality.md`：裁员协商过程中的情绪压力和现实成本。

## 贡献方式

欢迎有类似经历的人一起完善这个 Skill。

你可以：

1. Fork 这个仓库。
2. 基于 `main` 新建一个分支。
3. 修改或新增 `references/` 里的内容。
4. 提交 Pull Request。
5. 我会阅读后合并适合公开、可复用、已脱敏的内容。

适合贡献的内容包括：

- 新的 HR/ER 话术场景。
- 绩效申诉、PIP、调岗、考勤、背调、离职证明等真实经验。
- 更清晰的补偿计算例子。
- 经济性裁员、失业金、竞业限制等场景的公开规则和实务经验。
- 更稳妥的证据保全建议。
- 不同城市、不同公司类型下的流程差异。
- 律师咨询时值得问的问题。

## 脱敏要求

请不要在 PR 里提交：

- 公司名称、部门名称、真实人名、工号、手机号、邮箱。
- 未脱敏聊天截图、录音、合同、绩效材料。
- 具体到可以被内部人识别的日期、批次、组织架构、项目名称。
- 源代码、客户数据、内部系统截图、商业秘密。
- 情绪化攻击个人的内容。

请尽量把个人经历抽象成“可复用规则”。比如：

```text
不建议写：某年某月某日，某公司某 HR 对我说……
建议写：如果 HR 将签字与绩效结果绑定，应当要求公司提供书面依据，并保留不认可绩效结论的书面记录。
```
